# DownCollect 用例文档

本文档描述 DownCollect 卡牌游戏系统的所有用例，涵盖房间管理、游戏流程和信息展示。

---

## 1. 房间管理

### UC-1.1 创建房间

- **参与者**：玩家（房主）
- **前置条件**：玩家已打开游戏页面，系统已自动完成透明认证（Device Auth）
- **主要流程**：
  1. 玩家点击"创建房间"
  2. 系统生成唯一房间号（4-6位字母数字）
  3. 玩家自动加入该房间，成为房主
  4. 系统显示房间号和当前玩家列表
- **后置条件**：房间已创建，玩家已加入，等待更多玩家

### UC-1.2 加入房间

- **参与者**：玩家
- **前置条件**：玩家已打开游戏页面，系统已自动完成透明认证
- **主要流程**：
  1. 玩家输入房间号
  2. 系统验证房间号存在且游戏未开始
  3. 玩家加入房间
  4. 系统向所有房间内玩家广播新玩家加入
- **替代流程**：
  - 2a. 房间号不存在 → 提示"房间不存在"
  - 2b. 游戏已开始 → 提示"游戏已开始，无法加入"
  - 2c. 房间已满（4人）→ 提示"房间已满"
- **后置条件**：玩家已在房间中

### UC-1.3 设置玩家昵称

- **参与者**：玩家
- **前置条件**：玩家已打开游戏页面
- **主要流程**：
  1. 玩家在主页输入昵称（或使用系统生成的随机昵称）
  2. 昵称随玩家加入房间时同步给所有人
- **后置条件**：玩家拥有可显示的昵称

### UC-1.4 开始游戏

- **参与者**：房主
- **前置条件**：房间内有 2-4 名玩家
- **主要流程**：
  1. 房主点击"开始游戏"
  2. 系统校验玩家人数合法（2-4人）
  3. 系统初始化游戏状态，进入起始阶段
- **替代流程**：
  - 2a. 人数不足 → 提示"至少需要2名玩家"
- **后置条件**：游戏开始，进入起始阶段

### UC-1.5 玩家断线重连

- **参与者**：玩家
- **前置条件**：玩家之前在某房间中，因网络问题断线
- **主要流程**：
  1. 玩家重新打开页面或网络恢复
  2. 系统通过 localStorage 中的 device ID 重新认证
  3. 系统检测该玩家仍在某个游戏中
  4. 系统自动将玩家重连回游戏，推送当前游戏状态
- **后置条件**：玩家恢复至游戏中，看到当前状态

## 2. 起始阶段

### UC-2.1 角色选择

- **参与者**：所有玩家
- **前置条件**：游戏已初始化
- **主要流程**：
  1. 系统从角色池中为每位玩家随机抽取 2 个角色
  2. 每位玩家看到自己的 2 个角色候选（仅自己可见）
  3. 每位玩家选择其中 1 个角色
  4. 所有玩家选择完毕后，进入发牌阶段
- **超时处理**：超时未选择 → 系统随机为其选择一个
- **后置条件**：每位玩家拥有一个角色（当前为 Noop 实现）

### UC-2.2 游戏初始化发牌

- **参与者**：系统
- **前置条件**：所有玩家已选择角色
- **主要流程**：
  1. 系统从 54 张牌中随机移除 5 张
  2. 系统为每位玩家发 1 张手牌（仅本人可见）
  3. 剩余卡牌洗混形成公共牌堆
  4. 系统向所有玩家广播游戏初始状态
  5. 进入揭示阶段
- **后置条件**：牌堆就绪，每人 1 张手牌

## 3. 揭示阶段

### UC-3.1 揭示卡牌

- **参与者**：当前行动玩家
- **前置条件**：轮到该玩家行动
- **主要流程**：
  1. 系统提示当前玩家进行揭示
  2. 玩家点击"揭示"（或自动触发）
  3. 系统翻开牌堆顶牌，展示给所有玩家
  4. 系统根据卡牌效果类型决定下一步
- **后置条件**：卡牌已揭示，等待效果处理

### UC-3.2 处理卡牌效果

- **参与者**：当前行动玩家（部分效果需要玩家选择）
- **前置条件**：卡牌已揭示
- **主要流程**：根据卡牌类型分派处理
  - **自动效果**（无需玩家选择）：A, 2, 小Joker, 大Joker → 系统自动执行
  - **需要玩家选择位置**：7, 8 → 玩家选择当前行的插入位置
  - **需要玩家选择卡牌**：3（选两张暗置卡牌交换）, 5/6（选一张暗置卡牌替换）, 9（选一张暗置卡牌翻明）, 10（选一张暗置卡牌偷看）
  - **需要玩家选择行**：J（选一行翻转明暗）, Q（选一行展示后重洗低分牌）
  - **需要玩家选择多张卡牌**：K（每行各选一张）
  - **递归揭示**：4 → 加入手牌后触发再次揭示
- **超时处理**：超时未操作 → 系统随机选择合法选项
- **后置条件**：卡牌效果已执行，卡牌已放置

### UC-3.3 手牌交换

- **参与者**：当前行动玩家
- **前置条件**：卡牌效果处理完毕，玩家有手牌
- **主要流程**：
  1. 系统提示玩家是否交换手牌
  2. 玩家选择一张手牌与当前行最右侧卡牌交换，或选择不交换
  3. 若交换：手牌卡牌以暗置状态放入展示区，展示区卡牌加入手牌
- **替代流程**：
  - 玩家无手牌 → 跳过此步骤
- **超时处理**：超时 → 视为不交换
- **后置条件**：行动结束，轮转至下一位玩家

### UC-3.4 揭示阶段轮次管理

- **参与者**：系统
- **主要流程**：
  1. 每位玩家都行动一次后，当前轮次结束
  2. 下一轮起始玩家为上一轮起始玩家的下一位
  3. 重复直到完成 6 轮
  4. 进入选牌阶段
- **后置条件**：揭示阶段结束

## 4. 选牌阶段

### UC-4.1 选择卡牌

- **参与者**：当前行动玩家
- **前置条件**：轮到该玩家行动
- **主要流程**：
  1. 系统提示当前玩家选牌
  2. 玩家从卡牌展示区域的任意一行选择一张卡牌
  3. 选择的卡牌加入手牌最右侧
  4. 选择对其他玩家不可见
- **超时处理**：超时 → 系统随机选择一张
- **后置条件**：玩家获得一张新手牌

### UC-4.2 选牌阶段轮次管理

- **参与者**：系统
- **主要流程**：
  1. 每位玩家都行动一次后，当前轮次结束
  2. 下一轮起始玩家轮转
  3. 重复直到完成 4 轮
  4. 进入计分阶段
- **后置条件**：选牌阶段结束

---

## 5. 计分阶段

### UC-5.1 计分

- **参与者**：系统
- **前置条件**：选牌阶段结束
- **主要流程**：
  1. 系统对每位玩家同时计分
  2. 计分顺序：角色计分规则 → 私有计分规则 → 公共计分规则 → 卡牌特殊计分规则（右→左）
  3. 每张卡牌计分时只可见自身及左侧卡牌
  4. J/Q/K 触发时会跳过左侧所有卡牌的计分
  5. 系统汇总各玩家总分
- **后置条件**：所有玩家分数已计算

### UC-5.2 结算展示

- **参与者**：所有玩家
- **前置条件**：计分完成
- **主要流程**：
  1. 系统向所有玩家展示：
     - 每位玩家的手牌（全部翻开）
     - 每张卡牌的得分明细
     - 每位玩家的总分
     - 获胜者（分数最低者，可并列）
  2. 玩家可以查看详细的计分过程
- **后置条件**：游戏结束

## 6. 全局功能

### UC-6.1 透明认证

- **参与者**：系统
- **触发条件**：玩家首次打开页面或 session 过期
- **主要流程**：
  1. 系统检查 localStorage 中是否有 device ID
  2. 若无，生成一个 UUID 作为 device ID 并存储
  3. 使用 device ID 作为 WebSocket 连接的身份标识
  4. 后续所有请求使用该 device ID
- **后置条件**：玩家拥有有效 session

### UC-6.2 游戏状态同步

- **参与者**：系统
- **触发条件**：任何游戏状态变更
- **主要流程**：
  1. 核心引擎更新权威游戏状态
  2. 系统为每位玩家生成其可见视图（处理信息不对称）
  3. 通过 WebSocket 推送给各玩家
- **规则**：
  - 暗置卡牌：所有人看到"未知卡牌"
  - 明置卡牌：所有人看到具体卡牌
  - 手牌：仅本人可见
  - 偷看信息（10的效果）：仅操作者可见

### UC-6.3 操作超时处理

- **参与者**：系统
- **触发条件**：玩家在限定时间内未操作
- **主要流程**：
  1. 系统检测到超时
  2. 系统执行默认操作（随机选择合法选项 / 不交换）
  3. 游戏继续推进
- **后置条件**：游戏不因玩家不操作而卡住
